PracticeX_Crawler_Connectors_Spec
PracticeX Crawler Connectors Feature Spec
Problem Statement
Independent and group medical practices manage payer contracts, leases, BAAs, vendor agreements, employment documents, amendments, fee schedules, renewal notices, and related obligations across messy local folders and employee mailboxes. Important documents and renewal dates are often buried in attachments, shared drives, and ad hoc folder structures, which creates missed renewal windows, weak negotiation preparation, and poor operational follow-through.

PracticeX needs two first crawler connectors that can discover likely contract evidence and route it into the same governed ingestion pipeline:

Local Folder Discovery

Outlook Mailbox Connector

The demo implementation should feel production-shaped. The crawlers may be narrow in source coverage, but they should write to durable ingestion concepts: source connections, source objects, ingestion batches, ingestion jobs, candidate documents, audit events, and downstream review queues.

Goals
Enable a practice administrator to discover likely contract-related files from a local folder and Outlook mailbox.

Automatically ingest discovered high-signal items into the governed document ingestion pipeline.

Preserve source traceability for every discovered item through source objects, batches, jobs, and audit events.

Avoid silently converting discovered documents into canonical contract records without downstream review and classification.

Support a strong demo story: PracticeX can look across a messy folder and a real mailbox, identify contract evidence, deduplicate it, classify/import it as candidate material, and push it into a governed review workflow.

Non-Goals
No EHR integration in this phase.

No Zoom live connector in the first crawler release; Zoom transcripts can be imported through folder discovery if exported files exist.

No canonical contract creation without review or explicit downstream workflow.

No broad Drive, SharePoint, SFTP, Gmail, HRIS, or payer portal connectors in this slice.

No full legal AI risk review in the crawler itself; crawlers only discover, score, ingest, and route documents.

User Stories
As a practice administrator, I want to scan a local folder for likely contracts and renewal documents so that I can find important agreements without manually opening every file.

As a practice administrator, I want PracticeX to scan my Outlook mailbox for likely contract attachments so that agreements sent by payers, vendors, landlords, counsel, or staff are not missed.

As a compliance or operations leader, I want every discovered item to retain its original source metadata so that I can audit where a document came from.

As a reviewer, I want discovered documents to appear as candidate documents in the review queue so that I can confirm whether they are contracts, amendments, notices, or irrelevant files.

As a product demo user, I want the system to show what it found, why it matched, and what was ingested so that I trust the automation.

Product Principles
Connectors feed one ingestion pipeline.

Every discovered item becomes a source object.

Every ingestion run creates an ingestion batch.

Every imported item creates an ingestion job.

Automation can discover, score, and import evidence, but canonical contract records require downstream review.

PostgreSQL identifiers should remain unquoted snake_case.

The backend should remain a modular monolith with clear source connection, ingestion, contract, evidence, workflow, renewal, and audit boundaries.

Scope
Crawler 1: Local Folder Discovery
The local folder crawler scans a user-selected folder path or uploaded folder set and identifies files likely to contain contracts, renewals, obligations, or related administrative importance.

Supported first-pass file types:

PDF

DOCX

DOC

XLSX

TXT or exported transcript text where practical

Signals:

File name contains terms such as contract, agreement, amendment, renewal, fee schedule, lease, MSA, BAA, SOW, notice, termination, vendor, payer, reimbursement, employment, benefits, OSHA, HIPAA.

Folder path contains terms such as contracts, legal, leases, vendors, payers, HR, compliance, renewals.

Document text or metadata contains renewal date, effective date, expiration date, notice period, auto-renewal, term, payer, landlord, vendor, covered entity, business associate, fee schedule, rates, or reimbursement.

File modified date is within an optionally selected range.

Expected output:

Source object for every discovered item.

Candidate document for items passing minimum confidence.

Ingestion batch for each scan run.

Ingestion job for each imported candidate.

Audit events for scan start, scan completion, item discovered, item skipped, item imported, and item failed.

Crawler 2: Outlook Mailbox Connector
The Outlook crawler connects through OAuth with read-only mailbox access and searches for likely contract evidence in email subjects, senders, folders, dates, and attachments.

Supported first-pass attachment types:

PDF

DOCX

DOC

XLSX

Signals:

Subject contains contract, agreement, amendment, renewal, fee schedule, lease, MSA, BAA, SOW, notice, termination, payer, reimbursement, vendor, landlord, counsel, legal, benefits, employment, HIPAA, OSHA.

Attachment filename contains contract-like terms.

Sender or domain appears payer-like, vendor-like, legal/counsel-like, landlord/property-like, HR/benefits-like, or administratively relevant.

Optional mailbox folder scope such as Inbox, Legal, Contracts, Vendors, Payers, HR, Compliance, or selected folders.

Optional recent date range filter.

Message body contains renewal or contract terms.

Expected output:

Source object for every discovered email or attachment candidate, depending on current source model.

Candidate document for each attachment passing minimum confidence.

Ingestion batch for each mailbox scan.

Ingestion job for each imported attachment.

Audit events for OAuth connection, scan start, scan completion, attachment discovered, attachment skipped, attachment imported, and attachment failed.

Automation Boundary
The selected implementation mode is full automation for discovery and ingestion:

The crawler should automatically scan selected sources.

The crawler should automatically create source objects.

The crawler should automatically score discovered files and attachments.

The crawler should automatically import items above the configured threshold into the ingestion pipeline as candidate documents.

The crawler should not silently create canonical contract records.

The crawler should not mark review complete.

The crawler should preserve enough reason codes to explain why each item matched.

This balances demo automation with enterprise trust.

Requirements
P0 Must Have
Local folder crawler

Acceptance criteria:

Given a configured local folder source, when a scan is started, then the system creates an ingestion batch.

Given a folder containing supported file types, when the scan runs, then the crawler identifies supported files and records each as a source object or skipped source object with reason.

Given a file with contract-like filename, path, metadata, or content signals, when it crosses the confidence threshold, then the system creates an ingestion job and candidate document.

Given a file that fails processing, when the scan completes, then the failure is recorded without failing the entire batch.

Outlook mailbox crawler

Acceptance criteria:

Given a connected Outlook account, when a mailbox scan is started, then the system creates an ingestion batch for that run.

Given emails with supported attachments, when the scan runs, then the crawler evaluates subject, sender, date, folder, body snippet, and attachment metadata.

Given an attachment crossing the confidence threshold, when the scan processes it, then the system imports it into the ingestion pipeline as a candidate document.

Given an unsupported attachment type, when encountered, then the system records it as skipped with a clear reason.

Shared source and ingestion model

Acceptance criteria:

Given any crawler-discovered item, when it is recorded, then it is associated with a source connection and source object.

Given any imported item, when ingestion begins, then it is associated with an ingestion batch and ingestion job.

Given duplicate files or attachments, when detected by content hash or stable source identity, then the system avoids duplicate candidate documents and records deduplication status.

Review-safe automation

Acceptance criteria:

Given an imported candidate document, when ingestion completes, then it appears in the appropriate review queue or candidate list.

Given a high-confidence document, when imported, then it still does not become a canonical contract record without downstream review or explicit existing workflow.

Given a reviewer opens a candidate, then they can see source type, source object, discovered timestamp, confidence, and reason codes.

Audit and observability

Acceptance criteria:

Given any crawler run, when it starts and finishes, then audit events are recorded.

Given each item processed, when it is imported, skipped, deduplicated, or failed, then the system records a status and reason.

Given a scan completes, when the user views results, then they can see total discovered, imported, skipped, duplicate, and failed counts.

Demo results view

Acceptance criteria:

Retry failed jobs

Short answer
Yes, a standalone crawler is useful if we position it as a PracticeX Facility Discovery Agent.

It would run on the client’s machine or facility network, scan folders locally, prune aggressively, package only selected files, and upload only the optimized bundles to PracticeX.

faster scanning

Why browser-only is limited
A browser-based folder upload is good for demo and first-time use, but it has limits:

Browser can inspect file names and relative paths, but not deeply or consistently across all file types.

Large folder reads can be slow or memory-heavy.

Browser cannot easily scan mapped drives or network shares in a robust way.

Browser cannot run as a background process.

Browser ZIP creation can be heavy for thousands of files.

Browser cannot easily do resumable processing across days.

Browser security sandbox limits filesystem access.

If the tab closes, the scan stops.

It is hard to keep a durable local index of what was already scanned.

So for 50-200 files, browser flow is fine.

For 10,000 files across practice shared drives, a local agent is much better.

A facility-side crawler can reach those if the user has local access.

The web app cannot.

7. Local privacy controls

8. Cost controls
